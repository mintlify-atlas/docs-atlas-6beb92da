`ms-robo-admin` is the **sole owner** of the `uob_portal` database schema. All TypeORM entity definitions, migration files, and seed scripts live exclusively in that service. `ms-robo-web` is a schema consumer ‚Äî its `entities/` directory is a copy pushed from `ms-robo-admin` via a sync script and must never be edited by hand or used to generate migrations.

## Ownership model

```
ms-robo-admin (Schema Owner)          ms-robo-web (Consumer)
‚îú‚îÄ‚îÄ entities/  ‚óÑ‚îÄ‚îÄ source of truth    ‚îú‚îÄ‚îÄ entities/  ‚óÑ‚îÄ‚îÄ synced copy
‚îú‚îÄ‚îÄ migrations/  (13 files)           ‚îÇ   (no migrations ever)
‚îú‚îÄ‚îÄ seeds/                            ‚îî‚îÄ‚îÄ repositories/
‚îî‚îÄ‚îÄ repositories/
```

<Warning>
Never edit entity files inside `ms-robo-web/src/infrastructure/persistence/entities/` directly. Never run `bun run db:generate` or `bun run db:migrate` from `ms-robo-web`. All schema changes begin and end in `ms-robo-admin`.
</Warning>

## Schema propagation workflow

Use this workflow whenever you add a new entity, alter a column, or change a relationship.

<Steps>
  <Step title="Edit the entity in ms-robo-admin">
    Add or modify the TypeORM entity file under `ms-robo-admin/src/infrastructure/persistence/entities/`. Use the appropriate subdirectory ‚Äî `account/`, `portfolio/`, `identity/`, etc.

    ```bash
    # example: editing the portfolio entity
    # ms-robo-admin/src/infrastructure/persistence/entities/portfolio/portfolio.entity.ts
    ```
  </Step>

  <Step title="Generate the migration">
    TypeORM diffs your entity definitions against the current database schema and writes a new migration file into `src/infrastructure/persistence/migrations/`.

    ```bash
    cd ms-robo-admin
    bun run db:generate
    ```
  </Step>

  <Step title="Apply the migration to the local database">
    Run all pending migrations against the local PostgreSQL instance (port 5433, database `uob_portal`).

    ```bash
    bun run db:migrate
    ```

    To undo the last applied migration during development:

    ```bash
    bun run db:revert
    ```
  </Step>

  <Step title="Sync entities to ms-robo-web">
    Push the updated entity files from `ms-robo-admin` into `ms-robo-web`. This script copies files and does not touch migrations.

    ```bash
    bun run db:sync
    ```
  </Step>

  <Step title="Verify the sync">
    Run the dry-run check to confirm that the two `entities/` directories are identical. A clean output means no drift.

    ```bash
    bun run db:sync:check
    ```
  </Step>
</Steps>

If `db:sync:check` reports differences, re-run `db:sync`. Do not manually reconcile the files.

## Seeding reference data

Initial seed data (roles, instruments, system parameters, questionnaire questions, etc.) is loaded via dedicated seed scripts. Run them once after the first `db:migrate` on a fresh database.

```bash
cd ms-robo-admin
bun run seed:run
```

Seed modules are organised by domain under `src/infrastructure/persistence/seeds/`:

| Seed directory | Contents |
|---|---|
| `authorization/` | Roles and role-permissions |
| `user/` | Default admin user accounts |
| `instrument/` | Securities, currencies |
| `portfolio/` | Portfolio templates |
| `question/` | Questionnaire questions and answers |
| `organization/` | Companies, branches, agents |
| `field-property/` | UI field configuration |
| `profile-number/` | Sequence generators |
| `public-holiday/` | Public holiday calendar |

<Note>
`Portfolio` links to `User` via `user_id`, not to `Account`. This diverges from the original UTrade Robo handover spec. The change was introduced in migration `1773047833410-AddUserIdStatusIsPrimaryToPortfolio`. One user may have many portfolios; exactly one has `is_primary = true`.
</Note>

## Migration reference

All 13 migration files live in `ms-robo-admin/src/infrastructure/persistence/migrations/`. They are applied in timestamp order by `bun run db:migrate`.

| # | Migration file | Schema change |
|---|---|---|
| 1 | `1764699128757-migration.ts` | Initial schema ‚Äî core tables for users, accounts, clients, portfolios, questionnaires, instruments, holdings, and organization hierarchy |
| 2 | `1764701694294-migration.ts` | Add `company_id` and `contact_id` columns to `users`; backfill existing rows |
| 3 | `1764707702177-migration.ts` | Add unique constraint on `question_submissions.portfolio_id` |
| 4 | `1764732975783-migration.ts` | Rename `users.role` to `users.role_id`; extract `roles` table with `role_permissions` |
| 5 | `1766668366857-migration.ts` | Expand `instruments.asset_class` and `instruments.market` enums with new values |
| 6 | `1766672273158-migration.ts` | Create `goals` table (investment purpose, target amount, target year, withdrawal method) |
| 7 | `1769416876981-migration.ts` | Create `file_attachments` table with unique index on `key` |
| 8 | `1772142966005-migration.ts` | Create `tr_codes` and `tr_code_accounts` junction tables |
| 9 | `1772184717430-migration.ts` | Create `public_holidays` table with country/year/date indexes |
| 10 | `1773047833410-AddUserIdStatusIsPrimaryToPortfolio.ts` | Add `user_id`, `status`, and `is_primary` columns to `portfolios` |
