The client portal (`mfe-robo-web`) supports single sign-on via Keycloak using the OAuth 2.0 Authorization Code flow with PKCE (Proof Key for Code Exchange). This flow avoids transmitting secrets from the browser and includes CSRF protection via a state nonce. The Admin UI (`mfe-robo-admin`) does not support SSO — it uses username/password login only.

## Environment variables

Configure these three variables before the build. They are baked into the bundle at build time via the `VITE_` prefix.

```bash
VITE_KEYCLOAK_URL=https://keycloak-admin-rhsso.aws001nonprod.uobkayhian.com
VITE_KEYCLOAK_REALM=uobkh
VITE_KEYCLOAK_CLIENT_ID=robo-web-public
```

| Variable | Description |
|---|---|
| `VITE_KEYCLOAK_URL` | Base URL of the Keycloak server (no trailing slash) |
| `VITE_KEYCLOAK_REALM` | Keycloak realm name |
| `VITE_KEYCLOAK_CLIENT_ID` | Public client ID registered in the realm |

<Note>
The Keycloak client must be configured as a **public** client (no client secret) with the PKCE code challenge method set to `S256`.
</Note>

## PKCE utilities

Two utility functions in `src/utils/pkce.ts` handle the cryptographic requirements of PKCE:

```typescript
// Generate a random 64-character code verifier
export function generateCodeVerifier(): string {
  const array = new Uint8Array(64);
  crypto.getRandomValues(array);
  return Array.from(array, (b) => b.toString(36).padStart(2, '0'))
    .join('')
    .slice(0, 64);
}

// Hash the verifier with SHA-256 and encode as base64url
export async function generateCodeChallenge(verifier: string): Promise<string> {
  const encoder = new TextEncoder();
  const data = encoder.encode(verifier);
  const digest = await crypto.subtle.digest('SHA-256', data);
  return btoa(String.fromCharCode(...new Uint8Array(digest)))
    .replace(/\+/g, '-')
    .replace(/\//g, '_')
    .replaceAll('=', '');
}
```

Additional session helpers store and retrieve the verifier and state nonce during the redirect round-trip:

```typescript
// Persist verifier in sessionStorage before the redirect
export function saveCodeVerifier(verifier: string): void {
  sessionStorage.setItem('pkce_code_verifier', verifier);
}

// Read and atomically remove the verifier after the callback
export function getAndClearCodeVerifier(): string | null {
  const verifier = sessionStorage.getItem('pkce_code_verifier');
  sessionStorage.removeItem('pkce_code_verifier');
  return verifier;
}

// Encode the return path + nonce as a base64 state string
export function generateState(path: string, nonce: string): string {
  sessionStorage.setItem('pkce_state_nonce', nonce);
  return btoa(JSON.stringify({ nonce, path }));
}

// Verify the returned state matches the saved nonce
export function verifyAndGetState(stateParam: string): { path: string } | null {
  try {
    const { nonce, path } = JSON.parse(atob(stateParam));
    const savedNonce = sessionStorage.getItem('pkce_state_nonce');
    sessionStorage.removeItem('pkce_state_nonce');
    if (nonce !== savedNonce) return null;
    return { path };
  } catch {
    return null;
  }
}
```

## Authorization URL builder

`src/config/keycloak.ts` constructs the Keycloak authorization URL from environment variables:

```typescript
import { keycloakConfig } from '@/config/keycloak';

export function getAuthorizeUrl(
  codeChallenge: string,
  state: string,
  nonce: string,
  redirectUri: string,
): string {
  const params = new URLSearchParams({
    client_id: keycloakConfig.clientId,
    response_type: 'code',
    scope: 'openid',
    redirect_uri: redirectUri,
    code_challenge: codeChallenge,
    code_challenge_method: 'S256',
    state,
    nonce,
  });
  return `${keycloakConfig.url}/realms/${keycloakConfig.realm}/protocol/openid-connect/auth?${params}`;
}
```

The resulting URL has this shape:

```
https://<VITE_KEYCLOAK_URL>/realms/<VITE_KEYCLOAK_REALM>/protocol/openid-connect/auth
  ?client_id=robo-web-public
  &response_type=code
  &scope=openid
  &redirect_uri=https://portal.example.com/auth/callback
  &code_challenge=<base64url-sha256-of-verifier>
  &code_challenge_method=S256
  &state=<base64-encoded-nonce-and-path>
  &nonce=<random-32-char-string>
```

## Login flow (step by step)

<Steps>
  <Step title="User clicks Sign in with SSO">
    The login page renders a button that initiates the PKCE flow. Clicking it runs the following setup logic.
  </Step>
  <Step title="Generate PKCE parameters">
    Call `generateCodeVerifier()` to create a 64-character random string. Call `generateCodeChallenge(verifier)` to produce its SHA-256 base64url hash. Save the verifier with `saveCodeVerifier(verifier)` — it is stored in `sessionStorage` keyed as `pkce_code_verifier`.

    ```typescript
    const verifier = generateCodeVerifier();
    const challenge = await generateCodeChallenge(verifier);
    saveCodeVerifier(verifier);
    ```
  </Step>
  <Step title="Generate state nonce">
    Create a 32-character nonce with `generateNonce()`. Encode the current path and the nonce into a base64 state string with `generateState(currentPath, nonce)`. The nonce is saved in `sessionStorage` as `pkce_state_nonce`.

    ```typescript
    const nonce = generateNonce();
    const state = generateState(window.location.pathname, nonce);
    ```
  </Step>
  <Step title="Redirect to Keycloak">
    Build the authorization URL with `getAuthorizeUrl(challenge, state, nonce, redirectUri)` and navigate the browser to it. The `redirectUri` must match a URI registered in the Keycloak client configuration.

    ```typescript
    const redirectUri = `${window.location.origin}/auth/callback`;
    const url = getAuthorizeUrl(challenge, state, nonce, redirectUri);
    window.location.href = url;
    ```
  </Step>
  <Step title="User authenticates at Keycloak">
    Keycloak displays its login form. After the user authenticates successfully, Keycloak redirects back to `redirectUri` with `code` and `state` query parameters appended.
  </Step>
  <Step title="Handle the callback at /auth/callback">
    The `AuthCallbackPage` component reads the `code` and `state` query parameters from the URL. It calls `verifyAndGetState(stateParam)` to confirm the nonce matches — this guards against CSRF. Then it calls `getAndClearCodeVerifier()` to retrieve and remove the saved verifier.

    ```typescript
    const code = searchParams.get('code');
    const stateParam = searchParams.get('state');

    const stateData = verifyAndGetState(stateParam);
    if (!stateData) {
      // Redirect to login with error
      return;
    }

    const codeVerifier = getAndClearCodeVerifier();
    ```
  </Step>
  <Step title="Exchange the code for tokens">
    Call `POST /v1/auth/sso/callback` with the authorization code, the code verifier, and the redirect URI. The backend exchanges these with Keycloak and returns JWT tokens.

    ```typescript
    const redirectUri = `${window.location.origin}/auth/callback`;
    const data = await ssoCallback(code, codeVerifier, redirectUri);
    // data: { token, refreshToken, tokenExpires, user, idTokenHint? }
    ```

    Request body sent to the backend:

    ```json
    {
      "code": "abc123...",
      "codeVerifier": "x7f2k9...",
      "redirectUri": "https://portal.example.com/auth/callback"
    }
    ```

    Response:

    ```json
    {
      "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
      "refreshToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
      "tokenExpires": 1713484800,
      "user": { "id": "usr-00000001", "username": "testuser", "role": "USER", ... },
      "idTokenHint": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9..."
    }
    ```
  </Step>
  <Step title="Store tokens and redirect">
    Save the `idTokenHint` in `sessionStorage` (key: `sso_id_token_hint`) — you need it for SSO logout. Update the auth store with the user and tokens, then navigate to the original path from `stateData.path` or fall back to the dashboard.

    ```typescript
    if (data.idTokenHint) {
      sessionStorage.setItem('sso_id_token_hint', data.idTokenHint);
    }
    // Tokens are automatically stored in portal-accessToken / portal-refreshToken
    navigate(stateData.path || '/');
    ```
  </Step>
</Steps>

## Logout flow

SSO logout must terminate both the local session and the Keycloak session. Use `POST /v1/auth/sso/logout` with the stored `idTokenHint`:

```typescript
const idTokenHint = sessionStorage.getItem('sso_id_token_hint') ?? undefined;
const { logoutUrl } = await ssoLogout(idTokenHint);
// Clear local tokens
apiManager.clearTokens();
// Redirect to Keycloak logout endpoint to end the SSO session
if (logoutUrl) {
  window.location.href = logoutUrl;
}
```

The backend builds the Keycloak logout URL using `getLogoutUrl(idTokenHint, postLogoutRedirectUri)`:

```
https://<VITE_KEYCLOAK_URL>/realms/<VITE_KEYCLOAK_REALM>/protocol/openid-connect/logout
  ?id_token_hint=<idTokenHint>
  &post_logout_redirect_uri=https://portal.example.com/login
```

<Warning>
If you skip the `idTokenHint`, Keycloak may not end the SSO session. The user could re-authenticate without entering credentials on their next login attempt.
</Warning>

## Error handling

Keycloak appends an `error` parameter instead of `code` when authentication fails. The callback page checks for this before attempting the token exchange:

```typescript
const error = searchParams.get('error');
if (error) {
  const description = searchParams.get('error_description');
  // Redirect to login with error message
}
```

Common error values from Keycloak:

| `error` value | Cause |
|---|---|
| `access_denied` | User cancelled or was denied by Keycloak policy |
| `invalid_request` | Malformed authorization request |
| `temporarily_unavailable` | Keycloak server error |
